- desktop/ - Aplicação Electron principal
    - src/
        - main/
            - index.ts - O "Maestro" do processo principal. Ele não executa nada, mas direciona cada execução para seu devido local.
                * loadElectron: Carregamento do runtime
                * whenReady: É o sinal verde para prosseguir com qualquer item no electron
                * AppLifecycle: Gerencia o ciclo de vida da aplicação
                * ProcessSupervisor: Segurança dos processos filhos
                * SessionManager: Gerencia as sessões do usuário
                * CpuPool: Gestão do poder de processamento
                * WindowManager: Responsável pela criação e manipulação de janelas
                * DeepLinkHandler: Permite chamadas usando links externos (do mundo exterior)
                * lifecycle: Linha 41 até 46, ond eé feita toda gestão de desligamento. Sem isso os processos ficariam rodando mesmo após fechar o app.
                * Linha 51 até 56: Abertura da primeira janela + inicialização do servidor tRPC/IPC
                * Linha 61: Temos o catch para tratamento de erros fatais
            ----------
            - process/
                - health-monitor.ts - Responsável por vigiar os processos que rodam em segundo plano (Agentes de IA ou Indexadores). Se algo der errado, o kernel toma uma decisão antes que isso prejudique o sistema inteiro.
                    *  O evento envia um requestId para o processo e ele precisa responder com o mesmo id.
                    * timeoutMs: Verifica se o processo respondeu o ping dentro do tempo
                    * status: Verifica se o processo não retornou "degraded"
                    * memoryLimitMb: Define um limite de memória que o processo pode consumir
                    * unhealthyThreshold: Utiliza um contexto de `consecutiveFailures` pois o processo pode estar executando uma tarefa pesada e demorar.

                ----------
                - managed-process.ts - É a "baba" do processo.Ele cria, escuta, limpa e reinicia.
                    * pidusage: Usado para extrair informações de performance do processo
                    * restartCount: Evita que um processo seja recriado infinitamente
                    * waitForExit: Permite esperar um tempo para que o processo termine
                    * statusValue: Informa para a UI o verdadeiro estado do processo
                    * DisposableBase: Garante o descarte
                ----------
                - supervisor.ts - É o "CEO" dos processos. Enquanto o `ManagedProcess`, cuida de um, esse é o responsa'vel por vários. 
            ----------
            - services/
                - cpu-pool.ts - "Usina de Força". Resolve o problema de executar tudo em uma única thread.
                    * Cria de fato uma piscina de workers para que o kernel possa delegar o trabalho para processo secundários e continuar respondendo a UI.
                    * Sempre Subtrai -1 das CPUs, para exitar o consumo total da máquina.
                    * Threads extras são solicitadas apenas quando necessário.
                ----------
                - session-manager.ts - Garante que cada processo ou tarefa tenha seu próprio espaço seguro para rodar.
                    * Níveis:
                        - SessionManager: Nível mais alto, ele trabalha com ID de usuário e Sessão
                        - ProcessSupervisor: Intermediário, ele recebe o pedido e cria o processo.
                        - ManagedProcess: Baixo, cuida dos detalhes técnicos do processo.
                    * getOrSpawn: Se ja houver uma sessão ativa, ele retorna ela.
            ----------
            - workers/
                - cpu-pool/
                    - tasks.ts - Definição de tarefas que devem utilizar a piscina.
                        * parseJsonlFile: Fazer um parse de JSON pesados
                        * compressBuffer: Compressão de buffer
                        * renderMarkdownBatch: Formatar textos grandes
                        * Todas as funções recebem `args` pois threads separadas não compartilham memória. `pickString` garante a segurança de tipos entre as threads.
                ---------- 
                - session-worker.ts - É um serviço de longa duração no sistema.
                    * Funciona como um "mini-servidor" privado que existe para uma única sessão de tarefa. Nesse caso, um trabalho com mensagens.

            ----------
            - app-lifecycle.ts -  Garante o `graceful shutdown`. Aqui temos a garantia de que todos os processos são encerrados corretamente e o que não foi, é registrado em logs
                * shutdownHandlers: Fila de tarefas de limpeza
                * Promise.race: Garante que o fechamento não fique travado. Lançando logs de erro para o que não fechou.
                * preventDefault: Ganha tempo para salvar o estados antes de fechar
                * allWindowsClosed: Mantém a experiência nativa do SO (Mac vs Windows)

            ----------
            - deep-link-handler.ts - Um roteador de links especiais que não começam com `http://`
                * Evita links malformados ou perigosos
                * Verifica se há uma janela aberta e abre uma se não houver

            ----------
            - electron-runtime.ts - Extração de algumas tipagens do electron para garantir o funcionamento completo das implementações. É apenas um "dublê"

            ----------
            - ipc-bootstrap.ts - Apenas um arquivo de conexão entre o `ipc-server` e o `window-manager` para que o main principal permaneça pequeno

            ----------
            - ipc-context.ts - Arquivo que liga as features (dos packages) com as execuções reais do projeto
                * g4os/ipc/server: Aqui estamos usando todo o processo de contratos de validação criados no pacote de IPC.
                * neverthrow: Garante que o código não falhe e que a mensagem de erro é o mais clara possível.
                * generateTraceId: Permite rastrear toda a execução do usuário

            ----------
            - ipc-server.ts - Aqui é o momento em que conectamos o tRPC ao transporte IPC
                * createIPCHandler: Esse item quem "sequestra" as mensagens do IPC, Sempre que uma mensagem chega ele olha para o `AppRouter` e executa a procedure correspondente.
                * windows: options.windows - Apenas janelas conectadas tem permissão para enviar mensagens
                * loadElectronTrpc: Garante um código testável fora do electron
                * appRouter: Um mapa com todas as capas de IPC
                * Esse é o principal arquivo que "mata" a necessidade de criar "ipcMain.handle"
                * IpcInvokeEventLike: tipagem que evita sujar o código com as poluições do Electron

            ----------
            - window-manager.ts - Aqui temos a fabricação estrutural das janelas mas, não o conteúdo. Abre e fecha quando necessário.
                * contextIsolation: Garante que a UI não acesse o Node.js diretamente
                * nodeIntegration: Desativa a API do Node dentro da janela
                * sandbox: Isola a janela do SO o máximo possível
                * windows = new Set<BrowserWindowInstance>(); - Garante que uma janela não seja adicionada duas vezes e facilitada o calculo para obter o número de janelas.
                * this._register: Registro da função de descarte quando a janela é fechada
            ----------
            - gents-bootstrap.ts - Executa a inicialização do registro
            ----------
        - index.ts - Apenas uma fachada para o main/index.ts
        - preload.ts - Conexão entre o MAIN e o RENDERER
            * mod.exposeElectronTRPC(): Essa função está criando com canal. O código react utiliza o ipcLink para enviar mensagens, sabendo exatamente o canal.
