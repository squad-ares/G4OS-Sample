Code Review 39 — packages/kernel — 14 findings (1 CRITICAL + 5 MAJOR + 5 MEDIUM + 3 LOW)

Kernel é foundation, então a barra é alta: foram revisados Result/Disposable/fs-atomic/logger/schemas/types/errors/serialization/validation. ADRs consultados: 0002, 0010, 0011, 0012, 0013, 0032, 0050, 0060, 0153, 0159.

**F-CR39-1 — `MessageSchema.metadata.thinkingLevel` é enum hard-coded duplicado de `ThinkingLevelSchema` (CRITICAL — drift / catalog drift)**. `packages/kernel/src/schemas/message.schema.ts:67` declara `thinkingLevel: z.enum(['low', 'think', 'high', 'ultra']).optional()` inline. O schema canônico já existe em `packages/kernel/src/schemas/session.schema.ts:37` (`ThinkingLevelSchema`) — exatamente o tipo que o CR-30 F-CR30-2 unificou para acabar com o drift entre UI/agent. O fato de o schema de `Message.metadata` ter sido escrito com a mesma enum literal cria a mesma classe de bug que o CR-30 corrigiu: adicionar/renomear um nível em `ThinkingLevelSchema` (ex.: introduzir `'minimal'`) silenciosamente diverge — `Session.metadata.thinkingLevel` aceita o valor novo, `Message.metadata.thinkingLevel` rejeita por validação, persistindo `Message` sem o campo. Root cause: literal repetido em vez de referência de symbol. Fix: substituir por `ThinkingLevelSchema.optional()` (importar de `./session.schema.ts`). ADR-0153 (catalog drift) + ADR-0010 (event-sourced sessions, evita mismatch entre projection e replay).

**F-CR39-2 — `SystemMessageKindSchema` não é exportado, divergindo da intenção do ADR-0159 (MAJOR)**. `message.schema.ts:83` declara `systemKind: z.enum(['error', 'info', 'warning']).optional()` inline. CR-30 (em `code-review-30.md`) e ADR-0159 anunciam "SystemMessageKind enum em `@g4os/kernel/schemas/message.schema.ts`" — mas o enum não tem `SystemMessageKindSchema`/`SystemMessageKind` exportados, então qualquer caller (renderer, IPC, migration mapper V1→V2 do ADR-0159 §"Mapping V1 → V2") tem que repetir a literal `['error','info','warning']`. Já se vê o sintoma em `packages/features/src/chat/types.ts:44` (`readonly systemKind?: 'error' | 'info' | 'warning'` — tipo manual). Adicionar mais uma variante (`partner_help`, `plan` — explicitamente discutido como roadmap em ADR-0159 §Consequências) exigirá tocar em N lugares. Fix: extrair `export const SystemMessageKindSchema = z.enum(['error', 'info', 'warning']);` + `export type SystemMessageKind = z.infer<typeof SystemMessageKindSchema>;` e usar `SystemMessageKindSchema.optional()` no `metadata`. ADR-0159, ADR-0153.

**F-CR39-3 — Logger module-level singleton com transport eager viola "sideEffects: false" (MAJOR)**. `package.json:7` declara `"sideEffects": false`. `logger/logger.ts:66-86` cria `const baseLogger = pino({...})` no top-level do módulo — em dev (`NODE_ENV === 'development'`) também spawna um worker thread `pino-pretty`. Linha 143 cria `export const logger = wrapPinoLogger(baseLogger)` — singleton vivo. Consumers que importam só tipos (`@g4os/kernel/types`) acabam puxando todo o módulo `logger.ts` quando o barrel root é resolvido (porque `index.ts` faz `export *`). Resultado: (a) `sideEffects: false` é uma promessa quebrada — bundlers que tree-shake confiando no flag mantêm o estado certo só por sorte; (b) cada renderer/CLI que importa kernel paga o custo de inicializar pino + worker `pino-pretty` em dev mesmo sem nunca chamar `createLogger`. CR-30 já vinha mexendo no logger (guard `readProcessEnv` no renderer) mas não mudou a inicialização eager. Fix: lazy-initialize `baseLogger` via getter (function-scoped, memoizado) e consultá-lo em `createLogger` / `wrapPinoLogger`. Alternativa: marcar paths sensíveis em `package.json#sideEffects` como exceção. ADR-0060.

**F-CR39-4 — `truncateCauseChain` perde `stack` na cópia (MAJOR)**. `errors/app-error.ts:35-39` clona `cause` via `Object.create(prototype)` + `Reflect.ownKeys + getOwnPropertyDescriptor`. O `stack` do `Error` original é uma **own property** definida lazily pelo V8 quando lida pela primeira vez — se o stack ainda não foi acessado, `getOwnPropertyDescriptor(cause, 'stack')` retorna `undefined` no momento da cópia, e o clone fica sem stack. Pior: ao final do clone, `Error.captureStackTrace` não é chamado, então o `clone` virtual instance retornado nunca terá uma stack útil. O sintoma é silencioso em prod (Sentry/log mostra `cause.stack === undefined`) e só aparece em telemetria. Fix: forçar leitura do stack antes do clone (`void (cause as Error).stack`) e/ou copiar `stack`/`message`/`name` explicitamente. ADR-0011.

**F-CR39-5 — `bindToAbort` chama `disposable.dispose()` duas vezes quando signal aborta após bind (MAJOR)**. `disposable/helpers.ts:27-38`: o handler `onAbort` invoca `disposable.dispose()` (linha 32). Se posteriormente o caller chama `bound.dispose()`, o `toDisposable` interno chama `signal.removeEventListener` E `disposable.dispose()` novamente (linha 36). O `disposable` injetado pode não ser idempotente. O guard idiomático em VS Code é não chamar dispose no inner se já foi chamado pelo abort handler (flag local). Implementações próprias seguidas pelo padrão `IDisposable` em VS Code são tipicamente idempotentes (`#disposed` flag), mas o contrato de `IDisposable` em `disposable/types.ts` não exige idempotência — então passar um disposable cru pra `bindToAbort` é unsafe. Fix: rastrear flag local `disposed` e short-circuitar em ambos os branches. Ou exigir contrato de idempotência no `IDisposable`. Tests cobertos para signal pré-abortado e pós-explicit-dispose, mas não para `bind → abort → dispose`. ADR-0012.

**F-CR39-6 — `DisposableStore.add` lança após auto-dispose, deixando caller com objeto inacessível (MAJOR)**. `disposable/disposable-store.ts:11-23` faz `disposable.dispose()` (auto-cleanup) e em seguida `throw new Error(...)`. O caller que esperava receber `T` de volta (interface `add<T>(d: T): T`) recebe exception — mas o disposable que ele criou JÁ foi descartado. Em código de bootstrap onde o caller assumiu o disposable retornaria, isto causa NPE imediato + double-dispose se o caller também tentar `try/catch` e disposar manualmente. O comportamento V1/VS Code padrão é **logar warning** e descartar — não throw, porque store-disposed-during-add é tipicamente race de shutdown. Fix: trocar throw por `log.warn` (`@g4os/kernel/logger`) + retornar o disposable. Ou documentar em JSDoc explicitamente que `add` NÃO retorna em store-disposed (assinatura virou `T | never`). ADR-0012, ADR-0032 (graceful shutdown gera essa race naturalmente).

**F-CR39-7 — `validateAtomic` não é o único path; `deserializeJson` engole erro de parse (MEDIUM)**. `serialization/json.ts:18` faz `JSON.parse` num `try/catch` e em caso de falha chama `parseSchema(schema, undefined, ...)`. O erro de schema retornado descreve "expected X, got undefined" — perdeu completamente a posição/causa do `SyntaxError` original. Para JSONL com 10k linhas, debugging fica cego. Fix: emitir `AppError({ code: VALIDATION_ERROR, message: 'JSON parse failed', cause: syntaxError, context: { sample: raw.slice(0, 200) } })` direto em vez do roundtrip via `parseSchema(undefined)`. Adicionalmente: `deserializeJson*` é dead code — não tem callers em todo o monorepo (só o README do kernel cita) — considerar remover ou wirar em algum lugar (event store JSONL?). ADR-0011, ADR-0043 (event store JSONL).

**F-CR39-8 — `LegacyImportEntry` definido só como TS interface, sem schema Zod (MEDIUM)**. `types/project.ts:47-54` define `LegacyImportEntry` como `interface` manual. `LegacyProject` (linhas 38-45) também tem versão TS hand-written que duplica `LegacyProjectSchema` (project.schema.ts:108-116). Resultado: `packages/ipc/src/server/routers/projects-router.ts:20` recriou `LegacyImportEntrySchema` inline (z.object) — código rodando contra um shape definido em dois lugares. Drift garantido na próxima alteração. Já flagado em `code-review-38-ipc.md` como F-CR38-X (caps faltando) — kernel é a raiz do problema. Fix: definir `LegacyImportEntrySchema`/`LegacyProjectFileSchema` em `project.schema.ts` (com cap em strings, paridade com `CreateMcpStdioSourceInputSchema`), derivar tipos via `z.infer`, remover `interface` manual. ADR-0153.

**F-CR39-9 — `transport.ts` usa `as any` para cast em `pinoTransport` (MEDIUM)**. `logger/transport.ts:61-62` tem `biome-ignore lint/suspicious/noExplicitAny` + `as any`. Comentário diz que o tipo correto não bate com o shape estrito. Mesmo com a justificativa, é o único `any` em todo o kernel — viola ADR-0002 ("Zero `any`"). Padrão V1: definir um tipo helper `PinoTransportArg` com shape do que `pino.transport` aceita (Pino expõe `TransportTargetOptions`). Fix: importar `TransportTargetOptions` de `pino` e mapear `PinoTarget → TransportTargetOptions` antes de passar. Ou abrir ADR de exceção justificando o `any` (preferível: extrair). ADR-0002.

**F-CR39-10 — `tsconfig.json` exclui `__tests__` do typecheck (MEDIUM)**. `packages/kernel/tsconfig.json:10` faz `"exclude": ["src/**/*.test.ts", ..., "src/__tests__/**"]`. `pnpm typecheck` no kernel não vê tests; tests podem regrir silenciosamente (eg.: stale fixture refletindo schema obsoleto). O monorepo usa Vitest sem build separado, então tests podem rodar mesmo com type errors (Vitest tem seu próprio TS via vite-node). V1 também tinha esse padrão; em V2 a forçar function `npm typecheck` dá falsa sensação de cobertura. Fix: criar `tsconfig.test.json` que **inclui** tests (ou rodar `tsc --noEmit` separado em `__tests__`); usar no `test`/`test:watch` script. ADR-0002.

**F-CR39-11 — `MessageRoleSchema` não exporta type guard `isSystemError` mencionado no ADR-0159 (MEDIUM)**. ADR-0159 §Consequências/Negativas explicita: "Compensar com helper: `isSystemError(msg): boolean`". O helper não existe em kernel. Cada feature está reimplementando o check (`message.role === 'system' && message.systemKind === 'error'`) — visto em `packages/features/src/chat/components/transcript/message-card/message-card.tsx:49` e `system-message.tsx`. Fix: adicionar em `message.schema.ts` (ou `types/message.ts`):
```ts
export function isSystemError(msg: Pick<Message, 'role' | 'metadata'>): boolean {
  return msg.role === 'system' && msg.metadata?.systemKind === 'error';
}
```
ADR-0159.

**F-CR39-12 — `randomBytes` em `writeAtomic` sincrono pode bloquear event loop sob entropy starvation (LOW)**. `fs/atomic-write.ts:50` usa `randomBytes(8)` (sync). Em Linux com `/dev/random` rasa após boot ou containers fresh, `randomBytes` sync bloqueia ~ms. Para 1 escrita ocasional é negligenciável; para writes em ráfaga (event store JSONL append em cada tool result) cumulativo. Fix: trocar por `randomBytes(8, callback)` async ou `crypto.webcrypto.getRandomValues` (não bloqueia). ADR-0050.

**F-CR39-13 — `error-codes.ts` `ErrorCode` é `as const` object + `type ErrorCode = (typeof ErrorCode)[keyof typeof ErrorCode]` — viola `noEnum` da biome.json mas elegante (LOW)**. `biome.json:81` proíbe `enum` mas o padrão usado (`as const` + indexed lookup) é exatamente o substituto idiomático. Funciona. Observação: o pattern não é documentado como decisão em ADR; alguém querendo migrar pode tropeçar. Fix opcional: mencionar no `README.md` do kernel a convenção. ADR-0002.

**F-CR39-14 — `validation/env.ts` faz dupla leitura de `process.env[name]` em `getEnvOptional` (LOW)**. `validation/env.ts:23-25`: `getEnvOptional` lê `process.env[name]`, e se não-`undefined` chama `getEnv(name, schema)` que lê `process.env[name]` **outra vez**. Em prática env vars não mudam entre as duas reads (Node não emite signals em env), mas é trabalho duplicado. Race teórica num test que muta `process.env` entre as duas reads (improvável; vale o cleanup pra clareza). Fix: passar o `raw` direto pro `safeParse` em vez de reentrar em `getEnv`. ADR-0013.
