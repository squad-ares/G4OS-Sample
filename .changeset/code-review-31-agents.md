Code Review 31 — packages/agents — 14 findings (3 CRITICAL + 5 MAJOR + 3 MEDIUM + 3 LOW)

**F-CR31-1 — Codex stderr nunca drenado (CRITICAL)**. `apps/desktop/src/main/...` spawna Codex via `NodeSubprocessSpawner` em `packages/agents/src/codex/app-server/node-spawner.ts:12` com `stdio: ['pipe', 'pipe', 'pipe']` mas nada lê `child.stderr`. ADR-0072 explicita "stderr drenado" — qualquer log/warning escrito no stderr enche o pipe buffer (~64KB no Linux/macOS, 8KB no Windows) e o subprocess fica bloqueado em `write(2)` indefinidamente, congelando o turn em silêncio. Root cause: `wrapChildProcess` em `node-spawner.ts:17-75` só consome stdout. Fix: anexar listener `child.stderr?.on('data', chunk => log.debug(...))` (ou descartar para `/dev/null` via `stdio: ['pipe', 'pipe', 'ignore']`). ADR-0072.

**F-CR31-2 — CodexAgent não trata exit inesperado do subprocess (CRITICAL)**. `packages/agents/src/codex/codex-agent.ts:79-101` registra apenas listener de `'message'`. Se o subprocess Codex crashar mid-turn (OOM, signal externo, falha de bridge MCP), `AppServerClient.watchExit` (`client.ts:121-128`) emite o evento `'exit'` para `exitListeners`, mas o `CodexAgent` nunca se inscreveu nele. Resultado: o Observable do `run()` jamais completa nem emite `done`/`error` — `runToolLoop` no main fica pendurado até `drain` timeout. Fix: registrar `appServer.on('exit', ...)` dentro do `run()` que emita `error: AgentError.network('codex', { reason: 'subprocess_exit', code, signal })` + `done: 'error'` se o turno ainda não tiver completado. ADR-0072.

**F-CR31-3 — OpenAI completions trata `tool_calls` como `stop`, quebrando tool loop (CRITICAL)**. `packages/agents/src/openai/providers/completions.ts:101` faz `finishReason === 'length' || finishReason === 'stop' ? finishReason : 'stop'` — quando o modelo termina com `tool_calls` ou `content_filter`, o provider injeta `'stop'` no chunk `done`. O `OpenAIEventMapper.mapDone` (`event-mapper/event-mapper.ts:5-10`) então mapeia `'stop' → 'stop'` em vez de `'tool_use'`. Consequência: `runToolLoop` nunca detecta o sinal de tool use e termina o turno antes de executar o tool — toda chamada de função via OpenAI Chat Completions é silenciosamente abandonada. Fix: preservar `finishReason` original (`'tool_calls'` precisa atravessar o boundary do provider para o mapper coercer corretamente). ADR-0074.

**F-CR31-4 — Gemini multi-turn tool result tem `name='unknown'` (MAJOR)**. `packages/agents/src/google/config/mapper.ts:107` tenta extrair toolName via `(msg as unknown as { toolName?: string }).toolName ?? 'unknown'`, mas `Message` (`packages/kernel/src/schemas/message.schema.ts:45`) não tem campo `toolName` no nível da mensagem — só `tool_use.toolName` e `tool_result.toolUseId` nos blocks. Resultado: toda `functionResponse` enviada ao Gemini usa `name = 'g4_unknown_<hash>'`, que não corresponde a nenhuma `functionDeclaration` declarada — Gemini rejeita ou ignora silenciosamente, quebrando multi-turn tool use. Fix: rastrear o mapeamento `toolUseId → toolName` da assistant message anterior dentro de `mapMessagesToGemini` e olhar `block.toolUseId` no tool_result para resolver o nome. ADR-0075.

**F-CR31-5 — CodexAgent não cancela turn anterior em re-run (MAJOR)**. `packages/agents/src/codex/codex-agent.ts:71-72` faz `this.activeRequests.set(input.sessionId, requestId)` sem antes verificar/cancelar request anterior — sobrescreve o requestId silenciosamente. Os outros 3 agents (`claude-agent.ts:113-119`, `openai-agent.ts:123-129`, `google-agent.ts:123-129`) implementam `linkController` que aborta o controller anterior. Para Codex: se o usuário enviar dois turns rapidamente para a mesma sessão, o primeiro continua rodando no subprocess (tokens consumidos, eventos descartados via `event.requestId !== requestId` filter), enquanto a UI só recebe os eventos do segundo. Fix: em `run()` checar `activeRequests.get(sessionId)`; se existir, enviar `cancel` para o requestId antigo antes de assignar o novo. ADR-0072.

**F-CR31-6 — OpenAI completions nunca emite usage (MAJOR)**. `packages/agents/src/openai/providers/completions.ts:79-104` consome chunks da OpenAI sem habilitar `stream_options: { include_usage: true }` no request. O `OpenAIStreamParams` (`openai/types.ts:37-47`) também não declara o campo. Resultado: agent nunca emite `{ type: 'usage', input, output }` para Chat Completions API — observabilidade de tokens (`TurnTelemetry`, billing reconciliation, F-CR30-X usage tracking) fica zerada para todos os modelos OpenAI exceto o Responses path. Sibling Claude (`runner/event-mapper.ts:84-105`) e Google (`providers/genai-provider.ts:155-164`) emitem usage corretamente. Fix: adicionar `stream_options: { include_usage: true }` em `mapConfig` quando provider é completions; consumir `chunk.usage` em `translateRawChunk` e emitir `OpenAIStreamChunk` `'usage'`. ADR-0074.

**F-CR31-7 — OpenAI Responses emite `done` por cada tool call (MAJOR)**. `packages/agents/src/openai/providers/responses.ts:131-135` faz `if (event.type === 'response.output_item.done' && event.item?.type === 'function_call') yield { type: 'done', finishReason: 'tool_calls' }` — Responses API emite esse evento UMA VEZ POR TOOL CALL (não no fim do response). Quando o modelo chama 2+ tools no mesmo turn, o provider emite `done` no primeiro `output_item.done`, e o `OpenAIEventMapper.mapDone` finaliza o accumulator emitindo `tool_use_complete` para todos os tools acumulados, depois faz `done: tool_use`. O segundo `output_item.done` re-emite `done` + tenta finalizar accumulator vazio. `runToolLoop` desencadeia tool execution prematuramente com tool_use IDs que ainda não terminaram de receber `arguments`. Fix: só emitir `done` ao receber `response.completed` (já existe em :138); rastrear `output_item.done` apenas para fechar o accumulator daquela tool call específica. ADR-0074.

**F-CR31-8 — Google ignora `temperature` do AgentConfig (MAJOR)**. `packages/agents/src/google/config/mapper.ts:131-144` (`buildGeminiStreamParams`) recebe `config: AgentConfig` mas nunca propaga `config.temperature`. Nem `GeminiStreamParams` (`google/types.ts:16-23`) nem `GenAIProvider.openStream` aceitam temperature. SDK `@google/genai` aceita `generationConfig.temperature`. Resultado: usuário ajusta temperature na UI, valor é persistido na session, agents Claude/OpenAI honram, Google ignora silenciosamente. Fix: estender `GeminiStreamParams` com `temperature?: number`, propagar em `buildGeminiStreamParams`, anexar a `config.temperature` no `buildRequest` em `genai-provider.ts:128-138`. ADR-0075.

**F-CR31-9 — Codex `send.catch(subscriber.error)` viola contrato AgentEvent (MEDIUM)**. `packages/agents/src/codex/codex-agent.ts:111` faz `.catch((err: unknown) => subscriber.error(err))` quando o `appServer.send` inicial falha. Outros agents emitem `{type:'error', error}` + `{type:'done', reason:'error'}` antes de `subscriber.complete()` para preservar o contrato AgentEvent (caller espera `done` em todo turno). RxJS `subscriber.error` faz teardown sem `done` — `runToolLoop` precisa do `done:error` para finalizar a assistant message no event store (consistência com `finalizeAssistantMessage`). Fix: replicar o pattern dos outros agents — emitir error + done + complete em vez de propagar throw. ADR-0070, ADR-0072.

**F-CR31-10 — `BedrockProviderOptions` tem campos mortos (MEDIUM)**. `packages/agents/src/claude/providers/bedrock.ts:9-15` declara `region`, `accessKeyId`, `secretAccessKey`, `sessionToken` mas `BedrockProvider` (linhas 24-61) só consulta `options.sdkFactory`. Os outros campos nunca são lidos — sabemos que SigV4 runtime é construído fora via `sdkFactory` injetado, então o tipo é enganoso: caller acha que setar `region` afeta o cliente, mas só `sdkFactory` importa. Fix: ou remover campos mortos do tipo (force caller a injetar `sdkFactory` puro) ou aceitar `region`/credenciais e construir SigV4 runtime default no `loadDefaultSdk` quando `sdkFactory` ausente. Documentar no JSDoc qual é o caso. ADR-0071.

**F-CR31-11 — `OpenAIEventMapper.startedIndexes` é estado morto (MEDIUM)**. `packages/agents/src/openai/event-mapper/event-mapper.ts:14` declara `private readonly startedIndexes = new Set<number>()`; linha 46 adiciona índices nele; nenhuma leitura existe no arquivo nem em consumers. Set acumula sem GC enquanto o mapper viver — turnos com muitas tool calls acumulam memória sem efeito funcional. Fix: remover o campo (não é usado para de-dupe — `accumulator.has(chunk.index)` já cobre isso na linha 43). ADR-0074.

**F-CR31-12 — `BridgeMcpConnector` não implementa IDisposable (LOW)**. `packages/agents/src/codex/bridge-mcp/connect.ts:14-40` é classe com estado mutável (`handle`) mas não estende `DisposableBase` nem implementa `IDisposable`. `CodexAgent` chama `bridge.detach()` manualmente em `_register` (codex-agent.ts:63), mas o contrato `IDisposable.dispose()` (ADR-0012) é o padrão idiomático e bridge skeleton vai ganhar transports/sockets — fica inconsistente migrar depois. Fix: estender `DisposableBase`, registrar cleanup interno (timers de retry, sockets futuros) via `_register`, expor `dispose()` em vez de `detach()`. ADR-0012.

**F-CR31-13 — Streaming operators exportados mas inutilizados (LOW)**. `packages/agents/src/streaming/{batch-deltas,backpressure}.ts` exportam `batchTextDeltas(16)` e `dropIfBackpressured(100)`. ADR-0076 explicita: "`batchTextDeltas(16ms)` + `dropIfBackpressured(100)` como operadores RxJS compostos; text deltas coalescidos a ~60fps; eventos estruturais jamais descartados". Verificação `grep -rn "batchTextDeltas\|dropIfBackpressured" apps/desktop/src packages/session-runtime/src` retorna zero resultados — TurnDispatcher consome o Observable do agent direto sem aplicar os operators. Resultado: text_delta sob pressão não é coalescido (renderer recebe N eventos/seg em vez de ~60fps), e backpressure não dropa text_delta sob picos. Fix: aplicar `agent.run(input).pipe(batchTextDeltas(16), dropIfBackpressured(100))` no `runAgentIteration` em `@g4os/session-runtime` ou no `turn-dispatcher.ts`. ADR-0076.

**F-CR31-14 — `wrapAgentError` não diferencia 400/404/422 (LOW)**. `packages/agents/src/shared/errors/wrap-agent-error.ts:17-34` mapeia 401/403→invalidApiKey, 429→rateLimited, 5xx→network, resto→unavailable. Status 400 (bad request — request inválido, ex. modelo inexistente, parâmetro fora de range) e 422 (unprocessable) caem em `unavailable` que é UX confusa: usuário vê "Agent unavailable" para um bug do client. Fix: adicionar branch para `status === 400 || status === 422 → AgentError.invalidInput(...)` (já existe no kernel), e `status === 404 → AgentError.unavailable(provider, { reason: 'not_found' })` com mensagem específica. ADR-0011, ADR-0070.
